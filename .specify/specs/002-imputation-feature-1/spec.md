# Feature Specification: Data Profiling & Exploratory Report

**Feature Branch**: `001-data-profiling`
**Created**: 2026-04-02
**Status**: Draft
**Input**: User uploads a raw CSV file to Claude.ai

## User Scenarios & Testing *(mandatory)*

### User Story 1 — Generate Data Profile Report (Priority: P1)

A non-technical business user uploads a raw CSV to Claude.ai and wants to understand the state of their data before any cleaning happens. The system ingests the CSV, runs ydata-profiling to generate an HTML profile report, and then the LLM produces a natural language summary of the main insights — what stands out, what looks problematic, and what the potential impact is on downstream analysis. The natural language report may reference or embed key graphs from the ydata-profiling output to illustrate its findings.

**Why this priority**: Everything else in Skill A depends on understanding the data first. Without a profile, the LLM cannot make informed transformation decisions. This is the foundation of the entire pipeline.

**Independent Test**: Upload a raw CSV with known data quality issues (missing values, mixed types, duplicates). Verify that (1) an HTML profile report is generated by ydata-profiling, (2) the LLM produces a natural language report that accurately describes the key issues and their potential impact, and (3) the report references relevant graphs from the profiling output.

**Acceptance Scenarios**:

1. **Given** a user uploads a valid CSV with headers and at least two rows, **When** the system runs the profiling pipeline, **Then** an HTML profile report is generated by ydata-profiling and a natural language summary is produced by the LLM highlighting the main data quality issues and their impact.
2. **Given** a user uploads a valid CSV, **When** the profiling completes, **Then** the natural language report includes references to key graphs from the ydata-profiling output (e.g., distributions, missing value heatmaps, correlation matrices) to support its findings.
3. **Given** a user uploads a CSV with no data quality issues, **When** the profiling completes, **Then** the natural language report confirms the data is clean and highlights key characteristics of the dataset without fabricating problems.

---

### User Story 2 — Detect and Flag PII (Priority: P1)

During profiling, the system scans the dataset for potential PII (directly identifiable information such as names, emails, phone numbers, SSNs, addresses; indirectly identifiable information such as date of birth, zip code, job title; and financial information such as account numbers, credit card numbers, transaction details). The LLM flags any columns that may contain PII and issues a warning.

**Why this priority**: PII detection is a non-negotiable guardrail in the constitution. It must happen at the profiling stage before any transformations are applied. MVP uses open-source, non-sensitive datasets only, so this is a basic warning — but the detection must be present.

**Independent Test**: Upload a CSV containing columns with names, emails, and zip codes. Verify that the system flags these columns with a PII warning in the natural language report.

**Acceptance Scenarios**:

1. **Given** a CSV containing columns with direct PII (e.g., a column of email addresses), **When** profiling completes, **Then** the natural language report includes a warning identifying the column and the PII category (e.g., "Column 'email' may contain direct PII — proceed with caution").
2. **Given** a CSV with no PII, **When** profiling completes, **Then** no PII warnings are issued.

---

### User Story 3 — Handle Edge Cases at Ingestion (Priority: P1)

The system must handle malformed or edge-case CSV inputs gracefully at the profiling stage, before any transformations are attempted. This includes empty CSVs, single-row CSVs, duplicate column names, special characters in column names, and files without a valid tabular structure.

**Why this priority**: The constitution requires V1 to be a robust tool that handles edge cases, not a happy-path proof of concept. Catching these issues at ingestion prevents cascading failures in Feature 2.

**Independent Test**: Upload each edge-case CSV type individually and verify the system responds correctly (fails with a clear message, or processes with appropriate warnings).

**Acceptance Scenarios**:

1. **Given** an empty CSV (headers only, no rows), **When** the user uploads it, **Then** the system fails with a clear error message indicating no data was provided. No profiling report is generated.
2. **Given** a single-row CSV, **When** the user uploads it, **Then** the system processes the file and the natural language report includes a warning about statistical limitations (e.g., "This dataset has only one row — profiling statistics such as distributions and correlations are not meaningful").
3. **Given** a CSV with duplicate column names, **When** profiling runs, **Then** the natural language report flags the duplicates and identifies which columns are affected.
4. **Given** a CSV with special characters in column names (spaces, emojis, unicode), **When** profiling runs, **Then** the natural language report flags the non-standard column names.
5. **Given** a file that is not a valid tabular CSV, **When** the user uploads it, **Then** the system fails with a clear error message indicating the input is not a valid CSV.

---

### User Story 4 — Download Profiling Outputs (Priority: P2)

The user can download the HTML profile report generated by ydata-profiling and the natural language summary as a markdown file, so they can share or archive the profiling results independently of the cleaning pipeline.

**Why this priority**: Useful for auditability and sharing, but not required for the transformation pipeline to function. The critical dependency for Feature 2 is the internal natural language report, not the downloaded files.

**Independent Test**: Upload a CSV, run profiling, and verify that both the HTML report and the markdown summary are available for download from Claude.ai.

**Acceptance Scenarios**:

1. **Given** profiling has completed successfully, **When** the user requests the outputs, **Then** the HTML profile report and the natural language summary (markdown) are both available for download.

---

### Edge Cases

- What happens when the CSV contains only one column? The system should profile it and note the limitation in the natural language report.
- What happens when all values in a column are missing? The system should flag this in the profiling report and note the impact (e.g., "Column X has 100% missing values — this column provides no analytical value in its current state").
- What happens when the CSV contains mixed types in a single column (e.g., "123", "abc", "45.6")? The system should detect and report the mixed types during profiling.
- What happens when the CSV is extremely large and approaches Claude.ai session limits? The system should warn the user if the file size may affect profiling reliability.
- What happens when ydata-profiling fails or produces an incomplete report? The system should fail gracefully with a clear error message rather than proceeding with partial data.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: System MUST accept a single raw CSV file uploaded by the user to Claude.ai.
- **FR-002**: System MUST reject files that are not valid tabular CSVs with a clear error message.
- **FR-003**: System MUST reject empty CSVs (headers only, no rows) with a clear error message.
- **FR-004**: System MUST run ydata-profiling on the uploaded CSV and generate an HTML profile report.
- **FR-005**: The LLM MUST produce a natural language report based on the ydata-profiling output, summarizing the main insights and their potential impact on downstream analysis.
- **FR-006**: The natural language report MUST reference or embed key graphs from the ydata-profiling output (e.g., distributions, missing value patterns, correlations) to support its findings.
- **FR-007**: The natural language report MUST use plain language understandable to a non-technical stakeholder (Level B user). Basic statistical terms (mean, median, mode, outlier) are permitted without explanation. Method-specific terms MUST be explained on first use. All acronyms MUST be defined on first use.
- **FR-008**: System MUST scan the dataset for potential PII (direct, indirect, and financial) using a hybrid approach — LLM flags potential PII during analysis. MVP issues a basic warning only (e.g., "Column X may contain PII — proceed with caution").
- **FR-009**: System MUST detect and report duplicate column names during profiling.
- **FR-010**: System MUST detect and report special characters in column names (spaces, emojis, unicode) during profiling.
- **FR-011**: System MUST detect and report columns where all values are missing.
- **FR-012**: System MUST detect and report mixed types within a single column.
- **FR-013**: System MUST process single-row CSVs with appropriate warnings about statistical limitations.
- **FR-014**: System MUST make the HTML profile report and the natural language summary (markdown) available for download.
- **FR-015**: System MUST assign a unique version or run ID to each profiling execution.
- **FR-016**: Logs MUST never contain raw data values. All identifiers MUST be masked or hashed in any exported logs.

### Key Entities

- **Raw CSV**: The user-uploaded input file. Must have a valid tabular structure with headers. Single file only (no multi-file joins).
- **HTML Profile Report**: The ydata-profiling output. Contains statistical summaries, distributions, correlation matrices, missing value patterns, and visualizations.
- **Natural Language Report**: The LLM-generated exploratory summary based on the HTML profile. Written in plain language. Includes key insights, potential impact, PII warnings, and references to profiling graphs. This report is the input for Feature 2.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: Profiling completes and produces both an HTML report and a natural language summary for any valid CSV input.
- **SC-002**: The natural language report accurately identifies the top data quality issues present in the dataset (missing values, duplicates, type inconsistencies, outliers, column name problems). No fabricated issues.
- **SC-003**: PII warnings are raised for all columns containing direct PII (names, emails, phone numbers, SSNs, addresses) with zero false negatives on test datasets.
- **SC-004**: All edge cases (empty CSV, single-row CSV, duplicate column names, special characters, all-missing columns, mixed types) are handled correctly — either failing with a clear error or processing with appropriate warnings.
- **SC-005**: The natural language report passes the plain-language compliance check: no undefined acronyms, no unexplained method-specific terms, all metrics include context.
- **SC-006**: A non-technical stakeholder (Level B user) can read the natural language report and understand the state of the data without asking follow-up questions (stranger test).
